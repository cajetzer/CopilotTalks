## Part 1: Individual Persona Files

### Why Split — Three Real Reasons

**1. Machine Consumption Efficiency**
Agents currently pull 256 lines to retrieve one persona's ~40 lines. Individual files mean `#file:personas/sarah.md` delivers exactly what's needed. Importantly — `copilot-instructions.md` *already tries to do per-persona lookup* (`Sarah=ROI, David=expertise...`). Individual files give it the backing it's reaching for.

**2. Personas as Authoring QA Lenses** *(the emergent insight)*
Individual files transform personas from narrative characters into **quality gates during content creation**. Before publishing a tech-talk, run the relevant personas against it: Does this answer something Sarah would actually care about? Would David find the technical depth sufficient? This changes what personas *are*, not just where they live.

**3. Independent Arc Evolution**
Jessica in Module 0 is terrified. Jessica in Module 6 is confident. Without per-file arc tracking, agents can write the wrong Jessica. A `## Arc Notes (by module)` table in `jessica.md` solves this silently.

### The Architecture: Hub + Leaf

```
workshop/00-orientation/
├── PERSONAS.md              ← Hub: index, FanHub story arc, self-assessment
└── personas/
    ├── sarah.md
    ├── marcus.md
    ├── david.md
    ├── elena.md
    ├── rafael.md
    └── jessica.md
```

`PERSONAS.md` becomes lighter — retains the team table, story arc, and self-assessment. Each persona section becomes a 2-line summary + link to the leaf file.

**Individual file format:**
```yaml
---
status: active
updated: 2026-04-09
name: Sarah Chen
archetype: Skeptical Senior
experience: 15 years
lens: ROI, proof, measurable value
voice: Drily skeptical
quirk: "I've outlasted three 'revolutionary' tools this decade."
workshop-role: Sprint lead, standards enforcement
transformation: From "prove it" to "I'm convinced"
content-types: [workshop, tech-talk]
---
```
Below the frontmatter: full prose (Background, Goals, Fears, Voice in Exercises) + new `## Arc Notes (by module)` table.

**The one-file-per-persona rule:** Each persona has exactly one canonical source. Exercise files use personas *in context* but never redefine personality, voice, or arc. No metastasization.

## Part 2: Personas in Tech-Talks

### The Core Principle: "Sarah *Types*" Not "Sarah *Says*"

In workshops, personas are *characters* — dialogue, arcs, transformation. In tech-talks, they must be *archetypes* — the reader identifies with the *type* (skeptical senior who needs ROI proof), not the character in someone else's story.

`AGENTS.md` already encodes this: `Sarah=ROI, David=expertise, Marcus=workflow, Elena=quality, Rafael=business value`. That IS the tech-talk lens system — just not yet formalized or visible to readers.

### Three Concrete Patterns

**Pattern 1 — "Who This Hits" Box** *(top of talk)*
```markdown
> **🎯 Who This Hits Hardest:**
> If you evaluate every new tool against measurable ROI (Sarah), or need to know
> the architecture holds under real pressure (David) — this talk was written for you.
```
Rules: 2-3 personas max, one sentence each, placed after the opening question.

**Pattern 2 — Reaction Line** *(occasional, inline)*
```markdown
> 💬 *Sarah's question: "What's the actual time savings — not the theoretical ones?"*
```
Rules: Max 3 per talk, must surface a genuine concern, never longer than one line.

**Pattern 3 — Impact Table** *(in the Mental Model / transformation section)*
```markdown
| Archetype | Before | After |
|-----------|--------|-------|
| ROI-focused (Sarah) | Manual review: 45 min/PR | AI-assisted: 12 min/PR |
| Quality-driven (Elena) | Coverage gaps in edge cases | AI surfaces edge cases |
| Architecture (David) | Patterns enforced by convention | Patterns enforced by code |
```
Rules: 2-4 rows, archetype description first + name in parentheses, must include concrete metrics.

### What NOT to Do
- ❌ Scripted dialogue or conversation arcs — that's workshop territory
- ❌ Forcing all six personas — use only the 2-3 genuinely relevant to the talk
- ❌ Persona callouts that don't surface a real concern — noise, not signal
- ❌ Lead/Support structure — no main characters in tech-talks

## Part 3: The Novel Synthesis — Persona-as-Prompt

Persona files can simultaneously be documentation AND `.prompt.md` files.

`workshop/00-orientation/personas/sarah.prompt.md`

This single file is:
1. **A persona profile** — docs for humans and agents
2. **An invokable Copilot prompt** — author types `@sarah` to pressure-test any content
3. **A living example** — of the exact pattern taught in Module 3 (Custom Prompts) and Module 6 (Custom Agents)

The file ends with a prompt section:
```markdown
## Prompt Mode

When invoked as a prompt, evaluate content through Sarah's lens:
- Does this demonstrate measurable ROI?
- Are the metrics concrete (time saved, cycles reduced, errors eliminated)?
- Would a 15-year veteran with high standards find this credible?
- Is there hype without substance?

Respond in Sarah's voice: drily skeptical, won't fake enthusiasm, approval means something.
```

This dogfoods the curriculum — the training repo itself uses the patterns it teaches. Prototype with Sarah first.

## Why This Works

| Element | Rationale |
|---------|-----------|
| Hub+leaf architecture | Preserves human orientation experience; improves machine retrieval |
| YAML `content-types` frontmatter | Routes personas to appropriate content types |
| *"Sarah types not Sarah says"* | Prevents tech-talks from feeling like workshop spillover |
| Three constrained patterns | Concrete and immediately usable; prevents author choice paralysis |
| Author review rubric | Practical QA step: does this talk answer 2-3 persona concerns? |
| Persona-as-Prompt | Dual-purpose files that dogfood Module 3 & 6 curriculum |
| Graduated 3-phase adoption | De-risks the effort; start invisible, go visible after validation |
| Arc Notes per module | Solves the "which Jessica?" problem for agents generating Module 6+ content |
| One-file-per-persona rule | Prevents metastasization — the #1 structural risk |

**The key reframe:** Individual persona files aren't just an organizational improvement — they're **authoring infrastructure**. They turn personas from narrative characters into QA lenses, executable prompts, and content routing metadata.
